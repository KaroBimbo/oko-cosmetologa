# Анализ безопасности сервиса "Око косметолога"

Дата проверки: 7 июня 2026

## Что проверялось

- хранение API-ключей и переменных окружения;
- отсутствие секретов в Git-репозитории;
- серверный API route `POST /api/analyze`;
- подключение Supabase Auth;
- зависимости проекта через `npm audit`;
- тесты основных функций сервиса;
- production security headers на Vercel.

## Использованные инструменты

- Codex для статического анализа кода;
- `npm test` для проверки автоматических тестов;
- `npm audit --audit-level=moderate` для проверки зависимостей;
- `git grep` для поиска случайно закоммиченных секретов;
- проверка HTTP headers опубликованного сайта.

## Результаты проверки

### 1. API-ключи

Секретные ключи `APIFY_TOKEN` и `OPENROUTER_API_KEY` используются только на сервере в `app/api/analyze/route.js`.
В браузер они не передаются.

Файл `.env.local` добавлен в `.gitignore`, поэтому локальные ключи не попадают в GitHub.

`NEXT_PUBLIC_SUPABASE_URL` и `NEXT_PUBLIC_SUPABASE_ANON_KEY` являются публичными Supabase-переменными. Их можно использовать на клиенте.

Статус: хорошо.

### 2. Git-репозиторий

Поиск по репозиторию не обнаружил реальных Apify/OpenRouter/Supabase secret keys.
В репозитории есть только примеры переменных в `.env.example` и README.

Статус: хорошо.

### 3. Валидация входных данных

`POST /api/analyze` проверяет:

- город;
- услугу или препарат;
- лимит конкурентов;
- список Instagram-профилей;
- выбранные источники.

Лимит конкурентов ограничивается безопасным диапазоном, а внешние запросы имеют таймауты.

Статус: хорошо.

### 4. Supabase Auth

Добавлена регистрация/вход/выход пользователя через Supabase.
Если Supabase-переменные не заданы, интерфейс показывает понятное состояние вместо ошибки.

Статус: хорошо.

### 5. Зависимости

Команда `npm audit --audit-level=moderate` обнаружила умеренную уязвимость в цепочке `next -> postcss`.

На момент проверки установлена актуальная стабильная версия Next.js: `16.2.7`.
Автоматический `npm audit fix --force` применять нельзя, потому что он предлагает breaking-change сценарий и может сломать проект.

Решение: зафиксировать риск, следить за обновлением Next.js/PostCSS и обновить Next.js, когда выйдет безопасный стабильный патч.

Статус: принято как временный умеренный риск.

### 6. Security headers

Добавлены security headers через `next.config.mjs`:

- `Content-Security-Policy`;
- `X-Frame-Options: DENY`;
- `X-Content-Type-Options: nosniff`;
- `Referrer-Policy: strict-origin-when-cross-origin`;
- `Permissions-Policy`.

Это снижает риски XSS, clickjacking, MIME-sniffing и лишнего доступа к browser API.

Статус: исправлено.

## Тестирование

Автоматические тесты:

- проверяют payload формы анализа;
- проверяют обязательные server env keys;
- проверяют лимиты конкурентов;
- проверяют обработку Apify/OpenRouter ошибок и таймаутов;
- проверяют Supabase config и создание browser client.

Результат: `34/34` теста успешно.

## Основные риски и рекомендации

1. `POST /api/analyze` пока публичный. В будущем лучше ограничить запуск анализа авторизованными пользователями или добавить rate limit, чтобы не тратить API-бюджет на случайные запросы.
2. Токен Apify передается в URL запроса к Apify. Он не попадает в браузер, но теоретически может оказаться в server logs. Лучше заменить на авторизацию через header, если Apify actor endpoint это поддерживает.
3. Нужно периодически запускать `npm audit` и обновлять Next.js после выхода безопасного стабильного патча для PostCSS-цепочки.
